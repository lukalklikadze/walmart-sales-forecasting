## Validation Methodology

რანდომულ სპლიტებს ვერ გამოვიყენებთ time series ამოცანებისთვის, რადგან მომავლის ინფორმაციამ არ უნდა იქონიოს გავლენა წარსულისაზე. ამის გამო ჩვენ გამოვყავით train.csv-ის **ბოლო 39 კვირა** (2012-02-03 .. 2012-10-26) როგორც ვალიდაციის სეტი, ზუსტად იმავე ხანგრძლივობის, რაც არის რეალურ სატესტო დატაში. 

ორი წესი, რომელსაც ყველა მოდელი მიჰყვება:

1. **The Pipeline requirement.** თითოეული დალოგილი მოდელი არის სრული sklearn Pipeline, რომელიც აკეთებს ფრედიქშენს პირდაპირ RAW joined frame-დან. Inference-ის დროს manual preprocessing არ ხდება. მოდელებისთვის გვაქვს custom sklearn-თან თავსებადი wrapper-ები (`NBeatsRegressor`, `ProphetPerSeries`, `LGBMBoosterRegressor` და ა.შ.), ასე რომ ყველა მოდელი იტვირთება `mlflow.sklearn.load_model`-ის გამოყენებით და გამოიძახება `.predict(raw_df)`-ით.

2. **Two-stage training.** ჯერ მოდელი ტრენინგდება train_split-ზე და ფასდება ვალიდაციაზე. შემდეგ დალოგვამდე მოდელი თავიდან ტრენინგდება მთლიან train.csv-ზე. ეს იმიტომაა საჭირო, რომ სატესტო დატისთვის დაფრედიქთებაზე სატრენინგო დატის ბოლო 39-მა კვირამაც იქონიოს გავლენა.

## Feature Engineering

ყველაფერი რაც ამ ეტაპს ეხება მოთავსებულია `src/features.py`-ში როგორც sklearn transformer-ები, ორ ჯგუფად:

**Group A, `CalendarFeatures`** (გამოიყენება ყველა მოდელისთვის):

- `Year`, `Month`, ISO `WeekOfYear` და ციკლური `week_sin`/`week_cos`
- `IsHoliday` დაყოფილია ცალკეულ flag-ებად: `is_superbowl`, `is_laborday`, `is_thanksgiving`, `is_christmas_flagged`
- `is_week_before_christmas`: EDA-მ აჩვენა, რომ რეალური სფაიქი გაყიდვებში ფიქსირდება შობის კვირამდე **ერთი კვირით ადრე**. რადგან დასვენების კვირებს მეტრიკაში ხუთმაგი წონა აქვთ, ამ კვირის ცალკე ფიჩერად გამოყოფამ მნიშვნელოვანი როლი ითამაშა.
- MarkDown columns: MarkDown1-5 სვეტები 2011 წლის ნოემბრამდე საერთოდ არ არსებობს და შემდეგაც არასრულადაა ჩაწერილი (მონაცემების 64-74% აკლია). NaN-ის ნულებით ჩანაცვლება არ შეიძლება, რადგან ეს აურებს სიტუაციებს, სადაც მოცემული ფიჩერი მართლაც 0 იყო და სადაც არ იყო აღწერილი. ამისთვის ვაკეთებთ ორ რამეს: NaN-ებს ვცვლით 0-ით და ყოველ სვეტს ვუმატებთ `md{n}_recorded` indicator flag-ს (1 ნიშნავს, რომ მნიშვნელობა ჩაწერილი იყო, 0 რომ არა). 
- Store `Type`-ის One-hot encoding; `Size` და macro column-ები უცვლელი რჩება.

**Group B, `LagFeatures`** (მხოლოდ ხეებისთვის):

- `Weekly_Sales`-ის lag feature-ები თითოეული (Store, Dept) წყვილისთვის: მაგალითად, lag_52 არის ზუსტად 52 კვირით ადრინდელი გაყიდვები იმავე მაღაზიის იმავე დეპარტამენტში. ტესტირების დროს პრობლემა ისაა, რომ მომავალი კვირებისთვის წარსული გაყიდვები ცხრილში არ გვაქვს. ამის გამო `fit()` ინახავს მთელი training history-ის snapshot-ს (Store, Dept, Date, sales), ხოლო `transform()` ამ შენახული ისტორიით ითვლის lag-ებს ნებისმიერი შემომავალი row-სთვის, მათ შორის raw test row-ებისთვისაც. ყველა lag მხოლოდ ადრინდელ თარიღებს ითვალისწინებს, ასე რომ target-ის გაჟონვას თავიდან ვირიდებთ.

## Key Finding #1: the Lag-Horizon Trap


თავიდან ვგეგმავდით lag-ებს (1, 2, 52) და rolling statistics-ს (4 და 12 კვირის ფანჯრებით). 39 კვირიან ვალიდაციაზე ამ სეთაფმა მიიღო **WMAE ~10,153**, რაც უარესია იმაზეც კი, რასაც საერთოდ lag-ების გარეშე ვიღებთ (4,143).

ამის მიზეზი შემდეგი იყო: lag_1 ნიშნავს "წინა კვირის გაყიდვებს". ტრენინგის მონაცემებში ეს ყოველთვის ცნობილია, ამიტომ მოდელი ზედმეტად დამოკიდებული ხდება წინა კვირის დატაზე. რეალური პროგნოზისას 39 კვირით წინ ვიყურებით: მე-10 სატესტო კვირისთვის "წინა კვირა" მე-9 სატესტო კვირაა, რომლის გაყიდვებიც არაა ცნობილი. lag_1 და lag_2 სატესტო პერიოდისთვის თითქმის ყოველთვის ცარიელია (lag_1 შევსებულია სატესტო კვირების მხოლოდ 2.5%-ში). მოდელი, რომელიც ამ ფიჩერებზეა დამოკიდებული, პროგნოზისას მათ გარეშე რჩება და შედეგი მკვეთრად უარესდება.

ამის გასასწორებლად გადავწყვიტეთ მხოლოდ ისეთი lag-ები გამოვიყენოთ, რომლებიც ყოველთვის ცნობილ წარსულს უყურებენ. lag_52 ("იგივე კვირა შარშან") სატესტო კვირებისთვისაც ტრენინგის მონაცემებში ხვდება, ამიტომ ის თითქმის ყოველთვის შევსებულია (98.2%).

| Lag configuration | Val WMAE (39 კვირა) |
|---|---|
| lags=(1,2,52) + rolling(4,12) | 10,153 |
| lag-ების გარეშე (მხოლოდ Group A) | 4,143 |
| lags=(52,) | 2,143 |
| lags=(52,53) | 2,096 |
| **lags=(52,53,104)** | **2,096** |

დავასკვენით, რომ ამ მონაცემებში ყველაზე ძლიერი სიგნალი წლიური სეზონურობაა, ანუ ის, რაც ზუსტად ერთი წლის წინ, იმავე კვირაში მოხდა.

MLflow-ზე ორივე ვარიანტია დალოგილი, წარუმატებელი ცდაც ექსპერიმენტის ისტორიის ნაწილია.

## Key Finding #2: the Signed log1p Transform

გაყიდვების მნიშვნელობები ძალიან არათანაბრადაა განაწილებული: კვირების უმეტესობაში დეპარტამენტი რამდენიმე ათას დოლარს ყიდის, მაგრამ არის კვირები 100,000+ გაყიდვებითაც. ასეთ დროს ხშირად გვეხმარება ლოგარითმზე გადაყვანა (log1p), რომელიც ამ სხვაობას აპატარავებს.

პრობლემა ისაა, რომ 1,285 row-ში გაყიდვები უარყოფითია (დაბრუნებები აჭარბებდა გაყიდვებს), უარყოფით რიცხვზე კი ლოგარითმი არ არსებობს. ამიტომ გამოვიყენეთ ლოგარითმის ნიშნიანი ვერსია: `sign(y) * log1p(|y|)`. ტექნიკურად ეს `TransformedTargetRegressor`-ით კეთდება: მოდელი ტრენინგდება გარდაქმნილ target-ზე, ფრედიქშენი კი ავტომატურად უბრუნდება ჩვეულებრივ დოლარებს.

ამ გარდაქმნამ tree მოდელები გააუმჯობესა (XGBoost val WMAE 1849 → 1817), ამიტომ საბოლოო კონფიგურაციებში დარჩა.

## Models

პროექტში გავტესტეთ განსხვავებული სახის მოდელები: tree-based gradient boosting (XGBoost, LightGBM), კლასიკური სტატისტიკური მოდელები (Prophet, SARIMA) და deep learning არქიტექტურები (N-BEATS, DLinear, TFT). 

### XGBoost

XGBoost არის gradient boosting მოდელი: ის თანმიმდევრობით აშენებს გადაწყვეტილების ხეებს, სადაც ყოველი შემდეგი ხე წინა ხეების დაშვებულ შეცდომას ასწორებს. Time series სტრუქტურას ის თავისით ვერ ხედავს, ამიტომ დროითი ინფორმაცია feature-ების სახით უნდა მივაწოდოთ: სწორედ ამისთვისაა Group A + Group B. ჩვენი ამოცანისთვის მნიშვნელოვანია, რომ ის NaN მნიშვნელობებს ბუნებრივად ამუშავებს, რადგან LagFeatures ტესტის ნაწილზე ცარიელ მნიშვნელობებს წარმოქმნის.

Pipeline: `CalendarFeatures → LagFeatures → DropColumns → XGBRegressor`. საბაზისო პარამეტრები: `n_estimators=600`, `learning_rate=0.05`, `max_depth=8`, `subsample=0.9`, `colsample_bytree=0.9`, `tree_method="hist"`.

| Run | Val WMAE |
|---|---|
| baseline_lags (1,2,52 + rolling) | ~10,153 |
| submission_lags (52,53,104) | 1,849 |
| submission_lags_log1p | 1,817 |
| tuned_* საუკეთესო | **1,776.6** |

ჯერ ორივე lag კონფიგურაცია გავუშვი, რომ Lag-Horizon Trap-ის ეფექტი ზუსტად ამ მოდელზეც დაფიქსირებულიყო. შემდეგ გრძელ lag-ებზე დავამატე signed log1p ტრანსფორმაცია, რომელმაც შედეგი გააუმჯობესა (1,849 → 1,817). ასევე ვცადე დასვენების კვირების გაძლიერება ტრენინგშივე (sample_weight=5 დასვენების row-ებზე), რომ ტრენინგის მიზანი ზუსტად დამთხვეოდა WMAE მეტრიკას. ბოლოს ჩავატარე ჰიპერპარამეტრების sweep-ი (max_depth 6/10, n_estimators=1200 + learning_rate=0.03, min_child_weight=5 + subsample=0.8), საიდანაც საუკეთესო კომბინაციამ 1,776.6 მომცა.

Feature importance ანალიზმა EDA-ს დასკვნა დაადასტურა: ყველაზე მნიშვნელოვანი ფიჩერები აღმოჩნდა lag_52 (შარშანდელი იგივე კვირა), Size და დასვენების flag-ები. აქაც ფიჩერების დიზაინმა (გრძელი lag-ები) ბევრად მეტი მოგება მოიტანა, ვიდრე tuning-მა.

საბოლოო მოდელის მისაღებად გამარჯვებული კონფიგურაცია მთელ train-სეტზე გადავატრენინგე და დავლოგე როგორც სრული pipeline, რომელიც პირდაპირ raw test-ზე მუშაობს.

### Prophet

Prophet (Meta-ს მოდელი) დროით მწკრივს შლის ინტერპრეტირებად კომპონენტებად: `y(t) = trend + seasonality + holidays + noise`. მისი მთავარი ძლიერი მხარე ჩვენი ამოცანისთვის დღესასწაულების ჩაშენებული მხარდაჭერაა: მოდელს პირდაპირ გადაეცემა დღესასწაულების ცხრილი და ის თითოეულს ცალკე ეფექტს აძლევს. ეს ზუსტად ის თვისებაა, რაც ხუთმაგ წონიან მეტრიკაზე გვჭირდება.

Prophet ერთ მწკრივზე მუშაობს, ამიტომ დავწერე `ProphetPerSeries` wrapper-ი: ცალკე Prophet ყოველი (Store, Dept) წყვილისთვის, ჯამში ~3,300 მოდელი. დროში რომ ჩატეულიყო, ტრენინგი პარალელურად ეშვება joblib-ით და `uncertainty_samples=0` პარამეტრით, რადგან მხოლოდ point forecast გვჭირდება. მოდელს გადაეცემა დღესასწაულების ცხრილი, რომელშიც ოთხი ოფიციალური დღესასწაულის გარდა შესულია ჩვენი EDA-ს ნაპოვნი `week_before_christmas` კვირაც. მწკრივები, რომლებსაც 60 კვირაზე ნაკლები ისტორია აქვთ, სტაბილური fit-ისთვის საკმარისი არაა და series-ის საშუალო მნიშვნელობაზე გადადიან, ასე რომ submission-ში ცარიელი პროგნოზი ვერ ხვდება.

Ablation ჩავატარე დღესასწაულების ცხრილით და მის გარეშე: დღესასწაულებიანი ვერსია სტაბილურად უკეთესი იყო, რაც ადასტურებს, რომ ამ მეტრიკაზე holiday modeling-ის ხარისხი გადამწყვეტია.

საბოლოო ვერსია მთელ train-სეტზეა გადატრენინგებული და დალოგილია როგორც ერთი pipeline, რომელიც raw test-ზე პირდაპირ მუშაობს.

**Val WMAE: 1742.7**

### N-BEATS (GPU)

N-BEATS არის deep learning არქიტექტურა, აგებული fully-connected ბლოკების სტეკებად. თითოეული ბლოკი ორ რამეს გვაძლევს: backcast-ს (როგორ ხსნის უკვე ნანახ ისტორიას) და forecast-ს (რას ამბობს მომავალზე); შემდეგი ბლოკი მუშაობს იმაზე, რაც წინამ ვერ ახსნა. მას არანაირი ხელოვნური ფიჩერი არ სჭირდება: ქსელი პირდაპირ კითხულობს ბოლო კვირების raw ფანჯარას და თავად სწავლობს სეზონურ სტრუქტურას, ამიტომ Group B (LagFeatures) აქ საერთოდ არ გამოიყენება.

ეს არის ერთი გლობალური neuralforecast მოდელი, რომელიც ყველა ~3,300 series-ზე ერთდროულად ტრენინგდება (univariate, horizon h=39, GPU-ზე). გლობალურობა მთავარი უპირატესობაა: მოკლე ან ხმაურიანი მწკრივები სარგებლობენ პატერნებით, რომლებსაც ქსელი სხვა, უფრო მდიდარი მწკრივებიდან სწავლობს. Series-ები, რომლებიც `input_size + h`-ზე მოკლეა, ქსელის პროგნოზში ვერ ხვდებიან და series-ის საშუალოზე გადადიან; forecast coverage (მწკრივების რა წილს ფარავს ქსელი) იბეჭდება და მოწმდება notebook-შივე, რომ submission-ში NaN არ გაიპაროს.

Ablation ჩავატარე `input_size`-ზე: 52 კვირა (სრული წლიური ციკლი) და 26 კვირა. 52-კვირიანმა ფანჯარამ უკეთესი შედეგი დადო, რაც ლოგიკურია: ამ მონაცემებში მთავარი სიგნალი წლიური სეზონურობაა და მოდელს სრული წლის დანახვა სჭირდება, რომ ის დაიჭიროს.

საბოლოო კონფიგურაცია მთელ train-სეტზე გადავატრენინგე, რომ predict-ს ჰქონოდა უახლესი ისტორია და დავლოგე როგორც pipeline.

**საუკეთესო val WMAE: 1685.9**

### DLinear

DLinear (Decomposition Linear) არის time series პროგნოზირებისთვის გათვლილი მოდელი, რომელიც დაფუძნებულია სიგნალის დეკომპოზიციაზე. მონაცემები იყოფა ორ კომპონენტად: ტრენდად და სეზონურობად. ტრენდი არის გრძელვადიანი ტენდენცია, რომელსაც მოდელი გამოითვლის AvgPool1d-ს გამოყენებით, წინწასწარ განსაზღვრული, გადაცემული პერიოდისთვის. სეზონურობა კი არის ტენდენციებიდან გადახრა. მოდელი თითოეულ ფიჩერს ცალ-ცალკე შლის ტრენდად და სეზონურობად, თითოეულ კომპონენტს საკუთარი linear ფენა ამუშავებს, ბოლოს კი ორივეს შედეგი იკრიბება და საბოლოო პროგნოზი მიიღება.

ამ პროექტისთვის მას დამატებული აქვს პატარა cov_head MLP-ი, რომელიც მომავალი კალენდარული ფიჩერების (IsHoliday, is_thanksgiving, week_sin/cos და ა.შ.) გამოყენების საშუალებას აძლევს, რადგან ეს ფიჩერები წინასწარ ცნობილია, ანუ მაგალითად ჩვენ წინასწარვე ვიცით როდისაა მადლიერების დღე და ამ ინფორმაციის მოდელისთვის გადაცემა ინფორმაციის გაჟონვას არ გამოიწვევს.

DLinear-ს Group B-ს (LagFeatures) არ ვაწვდი, რადგან DLinear თავად სწავლობს დროით სტრუქტურას raw sales-ის გამოყენებით. სამაგიეროდ საჭირო გახდა DLinearTrainDataset-ის აწყობა, რომელიც sliding window-ებს ყველა სერიაზე აგროვებს ერთ pool-ში, რადგანაც ეს არქიტექტურა ერთი გლობალური მოდელია საზიარო წონებით ყველა სერიაზე, ანუ არა ~3300 ცალკეული პატარა მოდელი, არამედ ერთი ქსელი, რომელიც ერთდროულად ყველა store/dept-ის window-ზე ტრენინგდება ერთ batch-ში. ასევე დამუშავდა per-series z-score ნორმალიზაციით (mu, sigma)  ეს აუცილებელია, რადგან სხვადასხვა მაღაზია/განყოფილება რადიკალურად სხვადასხვა მასშტაბის გაყიდვებს აწარმოებს, და ერთი გლობალური მოდელისთვის სკალირების გარეშე ტრენინგი არასტაბილური იქნებოდა. ასევე დავამატე weighted_l1 loss-ი — იგივე 5x holiday-წონიანი ლოგიკა, რაც LightGBM-ის wmae feval-ში იყო, რათა ქსელმა ზუსტად იმავე მეტრიკაზე ისწავლოს, რომელზეც ფასდება.

აქაც კარგად ჩანს, რომ უფრო მოკლე lookback-მა (seq_len=39) აჯობა უფრო გრძელს (seq_len=65) რაც საინტერესოა, რადგან საწინააღმდეგოდ მუშაობს, ვიდრე LightGBM-ის lag-ისტორია. ეს იმიტომ ხდება, რომ Train-სეტში სულ 104 კვირაა ხელმისაწვდომი თითოეული სერიისთვის, რაც უფრო გრძელია seq_len, მით ნაკლები sliding window გამოდის თითოეული სერიისთვის სატრენინგოდ, და მოდელი ნაკლებ მაგალითზე სწავლობს. ასევე, DLinear-ის წრფივი ბუნების გამო, ძალიან გრძელ lookback-ს შესაძლოა ზედმეტი noise შემოაქვს სისტემაში, რომლის გაფილტვრასაც მარტივი წრფივი ტრანსფორმაცია ვერ ახერხებს ისე კარგად, როგორც tree-ს split-ები.

გამარჯვებული კონფიგურაცია (seq_len=39, kernel_size=13, lr=1e-3) სრულ 30 epoch-ზე გადავატრენინგე, შემდეგ მთელ train-სეტზე რეფიტი გავაკეთე საბოლოო ვალიდაციურმა შედეგმა 2348.95 WMAE მოგვცა, coverage 98.0%-ით, ხოლო რეალურ test-სეტზე coverage 98.8%-მდე გაიზარდა (ხანგრძლივი ისტორია სრულ raw-ზეც უფრო მეტ სერიას ფარავს). სერიები, რომლებსაც ქსელის პროგნოზი ვერ ფარავს, საშუალოზე გადადიან, ასე რომ submission-ში NaN პროგნოზი ვერ ხვდება.

### TFT

TFT (Temporal Fusion Transformer) არის time series პროგნოზირებისთვის აგებული  ერთი გლობალური ქსელი,რომელიც აერთიანებს LSTM-ზე დაფუძნებულ encoder-decoder არქიტექტურას static-ცვლადების (Store, Dept) ჩაშენებასთან და multi-head attention-თან. მას აქვს ოთხი ძირითადი ბლოკი:

- Static embedding 
- Encoder 
- Decoder 
- Attention 

ისევე როგორც DLinear-ის cov_head-ს, აქაც decoder-ს მიეწოდება მხოლოდ ის კალენდარული ფიჩერები (IsHoliday, is_superbowl, is_laborday, is_thanksgiving, is_christmas_flagged, is_week_before_christmas, week_sin/cos), რომლებიც წინასწარვე ცნობილია ანუ გაჟონვა არ წარმოიქმნება, რადგან ეს ინფორმაცია Date-ის სუფთა ფუნქციაა და test-სეტისთვისაც წინასწარვე გამოთვლადია.

Group B-ს (LagFeatures) TFT-სთვისაც არ ვიყენებ, იმავე მიზეზით, რაც DLinear-ის შემთხვევაში, deep net თავად სწავლობს დროით სტრუქტურას encoder-ის LSTM-ის საშუალებით raw sales-ის გამოყენებით, ცალკე lag-ფიჩერების გარეშე. ამიტომ საჭირო გახდა TFTTrainDataset-ის აწყობა: sliding (enc_len → pred_len) window-ები გროვდება ყველა (Store, Dept) სერიიდან ერთ pool-ში, რადგან ესეც ერთი გლობალური, საზიარო წონების მქონე მოდელია და არა ~3300 ცალკეული ქსელი. Window მხოლოდ მაშინ გენერირდება, თუ enc_len+pred_len თარიღები უწყვეტია, გამონაკლისებიანი სერიები გამოტოვებულია. ასევე გამოყენებულია per-series z-score ნორმალიზაცია (mu, sigma), fit()-ის დროს და შენახული predict-ისთვის, რათა store/dept-ებს შორის მასშტაბის განსხვავებამ ტრენინგი არ გააფუჭოს და დიდ გაყიდვებიან მაღაზიებს მეტი მნიშვნელობა არ მიანიჭოს, და იგივე weighted_l1 loss (holiday-კვირები 5x წონით), რაც LightGBM-ის wmae feval-ში და DLinear-ის loss-შიც იყო გამოყენებული იმისთვის, რომ ქსელმა ზუსტად იმ მეტრიკაზე ისწავლოს, რომელზეც ფასდება.

ჰიპერპარამეტრების ძიება ოთხ კონფიგურაციაზე ჩავატარე (enc_len, hidden, n_heads, learning_rate კომბინაციები), თითოეული 15 epoch-ზე დავატრენინგე ვალიდაციაზე შესადარებლად:

| კონფიგურაცია | val WMAE | coverage |
|---|---|---|
| enc_len=39, hidden=64, n_heads=4, lr=0.001 | **2051.19** | 98.0% |
| enc_len=52, hidden=96, n_heads=4, lr=0.0005 | 2146.83 | 97.9% |
| enc_len=52, hidden=64, n_heads=4, lr=0.001 | 2162.44 | 97.9% |
| enc_len=65, hidden=64, n_heads=2, lr=0.001 | 2613.15 | 97.6% |

აქაც იგივე ტენდენცია გამეორდა, რაც DLinear-ის შემთხვევაში, უფრო მოკლე lookback-მა (enc_len=39) აჯობა უფრო გრძელს (enc_len=65), რადგან Train-სეტში სულ 104 კვირაა ხელმისაწვდომი თითოეული სერიისთვის: რაც უფრო გრძელია enc_len, მით ნაკლები sliding window რჩება სატრენინგოდ და მოდელი ნაკლებ მაგალითზე სწავლობს. TFT-ის შემთხვევაში ამას ემატება ისიც, რომ გრძელი enc_len ნიშნავს LSTM encoder-ს დამატებით ნაბიჯებზე gradient-ის გავლას, რაც ასევე ართულებს კონვერგენციას მოკლე 15 ეპოქიანი sweep-ის ფარგლებში. hidden ზომის გაზრდამ 52/96 მცირედით გააუმჯობესა შედეგი 52/64-თან შედარებით, თუმცა ორივე ჩამორჩა 39/64-ს, ანუ ამ დატასეტის მასშტაბზე lookback-ის სიგრძე hidden-ის ზომაზე უფრო მნიშვნელოვანი აღმოჩნდა. ალბათ 39/64 -იც რომ მეცადა კიდევ უფრო კარგ შედეგს მომცემდა, მაგრამ მცირედით.

გამარჯვებული კონფიგურაცია (enc_len=39, hidden=64, n_heads=4, lr=1e-3) სრულ 30 epoch-ზე გადავატრენინგე raw_train_split-ზე, რამაც val WMAE 2010.47 მოგცა, coverage 98.0%-ით, სვიპის საუკეთესო შედეგზეც უკეთესი, ვინაიდან 30 epoch 15-თან შედარებით მეტ დროს აძლევს encoder-decoder-ს და attention-ს კონვერგენციისთვის. შემდეგ იგივე კონფიგურაცია მთელ train-სეტზე (raw, არა raw_train_split) თავიდან გადავატრენინგე, რათა predict()-ს ჰქონოდა სრული ისტორია, ამ საბოლოო pipeline-მა test-სეტზე coverage 98.8% აჩვენა. სერიები, რომლებსაც ქსელი ვერ ფარავს, აქაც საშუალოზე გადადიან, ასე რომ submission-ში NaN პროგნოზი ვერ ხვდება.

### SARIMA (theory write-up)

განზრახ არ არის დატრენინგებული; თეორიული განხილვაა.

ARIMA სამი კომპონენტისგან შედგება: AR (autoregressive: მიმდინარე მნიშვნელობა დამოკიდებულია წინა მნიშვნელობებზე), I (integrated: differencing ტრენდის მოსაშორებლად) და MA (moving average: დამოკიდებულება წინა შეცდომებზე). SARIMA ამატებს სეზონურ (P, D, Q, m) კომპონენტს, რომელიც იმავე ლოგიკას სეზონურ lag-ზე (ჩვენს შემთხვევაში m=52) იმეორებს. Notebook-ში ACF (autocorrelation function) ილუსტრაციით ნაჩვენებია, რომ ჩვენს მონაცემებს ზუსტად ის წლიური სეზონური სტრუქტურა აქვს, რომლისთვისაც SARIMA არის შექმნილი.

რატომ არ დავატრენინგეთ მოდელი: SARIMA ერთ მწკრივზე მუშაობს, ანუ დაგვჭირდებოდა ~3,300 ცალკეული fit, რაც გამოთვლითად ძალიან ძვირია; მწკრივებს შორის სწავლა არ ხდება (თითოეული მოდელი მხოლოდ თავის ~143 წერტილს ხედავს); მოსალოდნელი ხარისხი ჩამოუვარდება გლობალურ N-BEATS-ს, რომელიც იმავე უნივარიატულ ინფორმაციას ბევრად ეფექტურად იყენებს.
